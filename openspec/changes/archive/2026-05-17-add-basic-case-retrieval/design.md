## Context

DiagnoseToolPy has case storage and manual case creation, but no retrieval. When analyzing new incidents, similar historical cases cannot be found automatically. The retrieval system enables keyword and rule-based matching without requiring embedding models. BM25 is optional and only used when rank-bm25 is available.

## Goals / Non-Goals

**Goals:**
- Implement keyword search matching case content
- Implement rule matching by tags, components, fault_modes, exception_classes, key_phrases
- Support optional BM25 search when rank-bm25 is installed
- Generate prompt context with reference-only markers
- Keep retrieval working when embedding is disabled
- Independent from FastAPI and AI providers

**Non-Goals:**
- Do not implement mandatory embedding
- Do not implement mandatory vector database
- Do not call AI API
- Do not present historical root cause as confirmed current root cause

## Decisions

### Retrieval works without embeddings

Keyword and rule matching use only metadata (tags, components, fault_modes, exception_classes, key_phrases) and case content. BM25 adds full-text search when rank-bm25 is available. No embedding model required.

Alternative (embedding-first) was rejected because it would block deployment in environments without embedding infrastructure.

### Separate modules per retrieval channel

`keyword_search.py`, `rule_matcher.py`, `bm25_search.py` are separate to allow independent testing and optional enabling. `prompt_context.py` combines results for AI prompts.

Alternative (monolithic retrieval) was rejected for testability and maintainability.

### BM25 optional via dependency check

`bm25_search.py` checks if rank-bm25 is importable. If not available, BM25 search is skipped. This allows deployment without rank-bm25 while enabling it when available.

### Historical cases marked as references

Prompt context generated by `prompt_context.py` includes explicit markers stating cases are "references only" and "not assumed to have the same root cause". This follows the retrieval design requirement.

## Module Responsibilities

### `query_builder.py`
- `build_retrieval_query(analysis_result_or_path)`: builds query from retrieval-query.json or analyzer output path

### `keyword_search.py`
- `search_by_keywords(query, cases_dir)`: returns list of (case_id, score) tuples
- Scores based on keyword overlap with case content and metadata

### `rule_matcher.py`
- `match_by_rules(query, cases_dir)`: returns list of (case_id, score) tuples
- Scores based on tags, components, fault_modes, exception_classes, key_phrases overlap

### `bm25_search.py`
- `search_bm25(query, cases_dir)`: returns list of (case_id, score) tuples if rank-bm25 available
- Returns empty list if rank-bm25 not installed

### `prompt_context.py`
- `generate_prompt_context(query, cases, max_cases=3)`: generates markdown prompt context
- Includes reference-only markers

## File Outputs

No new files written. Retrieval operates on existing case files:
- `data/cases/{case_id}_{slug}/case.md`
- `data/cases/{case_id}_{slug}/metadata.yaml`
- `data/cases/index.yaml`

## Error Handling

- BM25 not available: log warning, skip BM25 search, continue with other channels
- Case directory missing: skip case, continue
- Malformed metadata.yaml: skip case, log warning
- Empty query: return empty results

## Security Considerations

- No network calls
- No code execution from case content
- Paths derived from case_id/slug, not user input

## Memory Behavior

- Streaming reads for large case.md files
- Bounded results (top-k per channel)
- BM25 corpus built lazily on first search
